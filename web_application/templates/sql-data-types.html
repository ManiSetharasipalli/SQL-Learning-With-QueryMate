SQL Data types

An SQL data type refers to the type of data that can be stored in a column of a database table.

By defining data types, users can specify the kind of information (numeric, string, binary, etc.) that can be stored in a column.

Examples of data types include: integer data, character data, monetary data, date and time data, binary strings, and so on.

Defining Table Columns

When creating a database table, two attributes are required to define a table column:

  • Column Name: A unique identifier for the column within the table.
  • Data Type: Specifies the type of data the column can store (e.g., integer, string, date).

Defining a Data Type

SQL Data types are defined during the creation of a table in a database. While creating a table, it is required to specify its respective data type and size along with the name of the column.

Following is the syntax to specify a data type in MySQL −

           CREATE TABLE table_name(column1 datatype, column2 datatype....)

Let us look at an example query below to understand better.

        CREATE TABLE Customers (Name VARCHAR (25), Age INT);

Types of SQL Data Types

There are three main types of SQL data types available in any RDBMS. They are listed below −

  • String
  • Numeric
  • Date and Time

Common SQL Data Types

Data Type Use Example RDBMS Support
INT Integer numbers Employee_ID INT All
VARCHAR Variable-length character strings Customer_name VARCHAR(50) All
DATE Date Order_date DATE All
FLOAT Floating-point numbers Product_price FLOAT All
BOOLEAN True/False values Is_active BOOLEAN MySQL, SQL Server, PostgreSQL, Oracle
BLOB Binary Large Object (large binary data) Product_image BLOB MySQL, PostgreSQL
CLOB Character Large Object (large text data) Product_description CLOB Oracle, PostgreSQL
DECIMAL Precise decimal numbers Product_weight DECIMAL(5,2) MySQL, SQL Server, PostgreSQL, Oracle
CHAR Fixed-length character string (padded with spaces) security_number CHAR(11) MySQL, SQL Server, PostgreSQL, Oracle

Note: Data types might have different names and details across different database systems. Always refer to the specific database documentation for accurate information.
